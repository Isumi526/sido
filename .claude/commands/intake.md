# /intake — 議事録を分解して「未整理」チケットを生成（出所リンク付き）。実装は /run へ

> **【正本（~/cc-pipeline）・プロジェクト中立版】** 固有値はハードコードしない。`{{BACKLOG_DS_ID}}`（バックログ管理 data source）/ `{{MINUTES_DS_ID}}`（議事録 data source）/ `.env` の `BACKLOG_PROJECT_ID`（自案件 page_id）等の `{{...}}` は各リポ CLAUDE.md「Pipeline設定」/`.env` 由来。通知は `scripts/notify-humanball.mjs`。

## これは何 / なぜ
タスク発生には2パターン：
- **① 単発依頼（LINE等）**：これまで通り「未整理」に1件追加 → /run が進める。/intake は不要。
- **② 会議・電話で複数かつ複雑に発生**：生の議事録を1つ投げ、そこから**複数タスクへ分解**する必要がある。これを担うのが /intake。

**設計の核**：/intake は **議事録 → "未整理" チケット生成まで**しか行わない（要件化・実装はしない）。分解した各タスクは **未整理**として作り、その後は**既存の /run §2 グルーミング（情報十分→要件定義済み／曖昧→要回答A/B/C）→ §3-4 実装**という実績のあるファネルに合流させる。＝分解ミスは未整理止まりで人が盤面で気づける安全弁を残す。

**厳守**：
- 生成するのは **未整理のみ**（要件定義済み/進行中に飛ばさない・実装しない・ブランチも切らない）。
- **1議事録=1案件**（議事録の `案件` を全生成チケットへ継承）。対象は `.env` の `BACKLOG_PROJECT_ID`（自案件）に一致する議事録だけ。
- **重複を作らない**：分解の各タスクは既存バックログ（自案件）を検索し、重複なら**新規作成せず既存にリンク**（+必要ならコメント）。
- 分解後に **抽出サマリを1回提示**（人が30秒で分解ミスをチェックできるように）。承認を待たず提示は必須（②は高判断のため）。

## §0 プリフライト（読み取りのみ）
1. **入力の特定**（引数 `$ARGUMENTS`）：
   - **空** … `{{MINUTES_DS_ID}}` の議事録のうち **`処理状態=未処理` かつ `案件`=自案件（`BACKLOG_PROJECT_ID`）** を古い順に対象化。0件なら「未処理の議事録なし」と報告して終了。
   - **議事録ページURL/ID** … その1件だけ。
   - **`--paste`** … チャットに貼られた議事録テキストを取り込む。まず §1 で議事録レコードを作成してから分解。
2. **自案件の確認**：`node --env-file=.env scripts/next-target.mjs --board` で現盤面（重複判定の材料）を把握。案件=`BACKLOG_PROJECT_ID`。

## §1 （--paste時のみ）議事録レコードを作成
チャット貼付テキストから議事録レコードを作る（両経路対応：Notion直追加でも良い）：
- `notion-create-pages`（parent=`{{MINUTES_DS_ID}}`）：`タイトル`（会議名や日付＋要旨）/ `日付`（分かれば）/ `案件`=自案件（relation）/ `出席者`（分かれば）/ `処理状態`=未処理。
  - **生の議事録テキストは `content`（ページ本文）に入れる**（長い議事録は見出し付きで読みやすい＝ここが raw ソースの正）。`本文`プロパティは任意（短い要旨を入れたい時だけ）。一覧プレビューに要旨を出したいなら `本文` に1行サマリを入れてもよい。
- 作成した議事録ページIDを以降の対象にする。

## §2 分解 → 未整理チケット生成（1議事録ずつ）
対象議事録を notion-fetch し `本文` を読む。**タスクだけを抽出**する（決定事項・FYI・雑談・既に完了済みの共有は除外／ただし「決定事項が新規作業を生む」なら抽出）。各抽出タスクごとに：

1. **重複チェック（必須）**：既存バックログ（自案件・完了/保留含む）に同義のものが無いか検索（`next-target.mjs --board` の結果＋必要なら `notion-query-data-sources` で `{{BACKLOG_DS_ID}}` をタイトル/内容検索）。
   - **重複あり** → 新規作成しない。その既存チケットを議事録の `生成チケット` にリンクし、必要なら「(議事録Xで再度言及: <要旨>)」を本文へ追記。サマリに「既存にリンク」と記録。
   - **重複なし** → 次へ。
2. **未整理チケット作成**（`notion-create-pages`・parent=`{{BACKLOG_DS_ID}}`）：
   - `タスク名`：命令形で簡潔に。
   - `ステータス`=**未整理**。
   - `案件名`=議事録の案件（継承）。
   - `生成チケット`側の逆リンク：作成したチケットを議事録の `生成チケット` relation に追加（＝出所が双方向で辿れる）。
   - 本文：議事録から拾った**元発言/文脈を1〜3行**＋「出所: 議事録『<タイトル>』(<url>)」。※ここでストーリー/AC/リスク等は**付けない**（/run §2 がやる）。
   - **優先度/リスク/エピック等は付けない**（未整理のまま /run §2 に委ねる）。ただし議事録に緊急語（障害/業務停止/至急）が明確なら `優先順位=緊急` だけは付けてよい（/run 緊急レーンに乗る）。
3. 抽出タスクを全部処理したら次の議事録へ。

## §3 議事録を処理済みに + 抽出サマリ提示
- 対象議事録の `処理状態`=**処理済み** に更新（REST PATCH で確実に・再確認）。
- **抽出サマリを提示**（結論ファースト）：
```
━━━━━━━━━ 📥 intake: 議事録「<タイトル>」(案件=<自案件>)
新規 未整理 N件:
  1. <タスク名>
  2. …
既存にリンク M件（重複回避）:
  - <既存タスク名>
除外（タスクでない=決定/FYI/雑談）: K件（要点だけ）
━━━━━━━━━
```
- 分解に自信が無い/タスクか判断が割れる項目は、サマリに「⚠要判断」で挙げる（未整理としては作っておき、/run §2 が要回答化する）。

## §4 ハンドオフ（実装はしない）
- 「未整理を N件作成。**次は /run で実装**（優先順位順・要件化 or 要回答は /run §2 が判定）」と案内。
- **/intake は実装・要件化・ブランチ作成・push を一切しない**。
- 複数議事録があれば §2〜§3 を繰り返し、最後に全体サマリ。

## 通知
- 未整理を生成してハンドオフする時に1通（best-effort）：
  `node scripts/notify-humanball.mjs --kind 要対応 --task "intake: 議事録<タイトル>" --detail "未整理N件生成(既存リンクM件)。/run で実装可。分解確認は議事録ページ" [--url "<session>"]`
- ⚠要判断の抽出があれば `--kind 要回答` に切り替え、その旨を detail に。

## DRY_RUN（`--dry-run`）
- 対象議事録・抽出予定タスク（新規/既存リンクの別）・付与予定（案件継承・緊急のみ例外）を**提示のみ**。Notion作成/更新・通知はしない。
