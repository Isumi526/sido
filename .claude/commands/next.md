# 盤面の確認（自動列挙は信用しない）

- Notionの全件列挙はしない。notion-search は曖昧検索で、要件定義済み/レビュー待ちの行を平気で取りこぼす。件数を断定するな。「○件ある」とカウントして、それを根拠に対象を選ぶのは禁止。
- 盤面の真実は Notion側のビューにある：
  - Readyキュー = 要件定義済み（次に着手できる・優先順位順）
  - 要対応 = レビュー待ち / 本番待ち（人の確認・出荷待ち）
- 本番反映が必要なものは /ship の役割。/next では扱わない（案内のみ）。

# 対象の決定（必ず人が渡す）

## $ARGUMENTS が渡された場合（NotionページのURL / ID / タスク名）
1. notion-fetch で そのページをURL/IDで直接取得する（検索は使わない＝確実）。
2. 取得ページの ステータス で分岐：
   - 要件定義済み → 【設計→実装】へ
   - レビュー待ち → 【レビュー→dev統合】へ
   - 本番待ち → 本番反映は /ship。ここでは案内のみ。
   - それ以外（未整理 / 進行中 / 保留 / 完了 等）→ 現状を報告して停止。

## $ARGUMENTS が空の場合
- 自動で対象を選ばない。推測して進めるのは禁止。
- 次のとおり人に促して停止し、URLが渡されるまで待つ：
  「盤面の全件はNotionのビューが正です。Readyキュー（要件定義済み）か要対応（レビュー待ち/本番待ち）を開いて、着手するタスクのページURLを /next <URL> で渡してください。優先順位はReadyキューの並び（緊急 > 高 > 中 > 低）で判断を。」

# ステータス=要件定義済み →【設計→実装】

1. 要件（背景/目的・ユーザーストーリー・機能要件・AC・対象範囲・制約）取得。曖昧点は実装前に質問。

2. Notion→「設計中」。

3. リポジトリ調査（CLAUDE.md・関連コード・スキーマ・edge functions/admin/liff）。

4. 詳細設計書を docs/design/{id}-{slug}.md に生成（データモデル/migration[追加のみ・後方互換明記]、

   edge functions変更[_shared import追跡]、admin/liff UI、AC→E2Eケース、デプロイ順序・後方互換性）。

5. ★停止。設計要点を提示し承認を待つ。

6. 承認後: 現在のdevを基点に feature/{id}-{slug} を切り実装（タスク毎コミット）。

7. AC対応のPlaywright E2Eを書きローカルで全green。

8. 報告＋Notion→「レビュー待ち」、設計書リンクをNotionに貼る。

# ステータス=レビュー待ち →【レビュー→dev統合】

1. featureブランチ特定（Notionの「ブランチ」or 引数）。未コミット変更があれば先にコミット。

2. レビュー材料提示: git diff --stat、主要変更点の要約、docs/designの要点、E2E結果、後方互換/DB変更有無。

   そのうえで「devにマージしてよいか」を尋ねて停止。

3. 「OK」なら: git switch dev && git merge {branch} --no-edit → 用済みブランチ削除 → Notion→「本番待ち」→ 報告。

   「NG」/修正指示なら実装して再E2E→再提示。

# それ以外のステータス

何もせず現状だけ報告。

# 共通：残課題の自動キャプチャ（全フェーズ）

作業中に「今回のスコープ外だが対応すべき」派生課題・技術的負債・既知の不具合を見つけたら、

Notionバックログに新規「未整理」として自動登録する。ただし：

- 登録前に notion-search で同様タスクが無いか確認（重複回避）。

- 着手価値のある具体的な課題のみ（自明・些末は登録せず報告に留める）。

- 薄いストーリー形式：タイトル＝課題が分かる一文／案件=SIDO／ステータス=未整理／タグ適切に／スコープ該当／

  本文に「## メモ」と「## 受け入れ条件(AC)」の雛形。RLS変更や設計判断が要るものはその旨明記。

- 登録したら「新規登録した未整理タスク」を一覧報告。
