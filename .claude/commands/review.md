# /review — レビュー待ちを優先順位順に人力テストをステップ・ナビ → 承認で本番待ちへ

`/run` がレビュー待ちに積んだタスクを、**人間が１件ずつ手で確認するための対話ナビ**。各タスク本文の「🧪人力チェック手順」を**1ステップずつ提示**し、**毎ステップにテスト用URL・アカウント・期待結果を明示**して、人間の「OK/NG」を受けて進める。全ステップOKなら**レビュー承認＝ステータスをレビュー待ち→本番待ちに反映**する（CLAUDE.mdの唯一の人間ゲート「レビュー待ち→本番待ち昇格」を、この対話を通じて人間が実行する）。

> **大前提**: レビュー待ちのコードは **dev にマージ済みだが未push＝本番には出ていない**（`vercel.json` でdevデプロイ無効）。だから人力テストは原則 **ローカルでdevブランチを動かして**行う。本番URLでの確認は ship 後（/ship）。本コマンドは「本番に出す前の最終人間チェック」。
> 承認後の本番反映は **/ship**（別の人間ゲート）。本コマンドは push / deploy / ship をしない。

引数 `$ARGUMENTS`:
- 空 … 自案件のレビュー待ちを**優先順位順**（緊急>高>中>低、同率は作成日古い順）に上から順にナビ。
- NotionページURL/ID/タスク名 … その1件だけをナビ。
- `--list` … レビュー待ち一覧（優先度順・リスク・エピック）を出すだけで終了（ナビしない）。

---

## §0 プリフライト（1回）

1. **対象取得**: `node scripts/next-target.mjs --board` を実行し、**「レビュー待ち」グループ**を読む（既に優先度順＋作成古い順にソート済み・各行は `[優先順位/リスク🧱土台]` 形式でリスクも表示される）。`$ARGUMENTS` 指定時は notion-fetch で直接取得。0件なら「レビュー待ちなし」を報告して終了。
2. **ローカル環境を起動**（devコードをローカルで触るため）:
   - Supabaseローカル: 落ちていれば `supabase start`。
   - admin: http://localhost:3001/ （未起動なら `cd apps/admin && npm run dev`）
   - liff: http://localhost:3000/ （未起動なら `cd apps/liff && npm run dev`）— LIFFは認証スキップ・テスターユーザーで動作（CLAUDE.md）。
   - **local接続assert**（本番を誤って触らない）: 配信anonキーが local の `sb_publishable…` か確認（`curl -s localhost:3000/ | grep -oE 'sb_publishable_[A-Za-z0-9]{6}|eyJ[A-Za-z0-9]{6}'`）。`eyJ…`（本番JWT）を検出したら停止し `.env.local` で起動し直す。
   - **ブランチ確認**: レビュー対象のコードが今のローカルに居るか。基本 `dev` を見る（`git branch --show-current` が dev か、無ければ `git checkout dev`）。作業中の差分があれば退避を促す。
3. **一覧プレビュー**: レビュー待ちを `① [高/🔴] タスク名（エピック）` の形で番号付き提示。`--list` ならここで終了。

---

## §1 各タスクのナビ開始（上から1件ずつ）

対象ページを notion-fetch し、本文の **「📋レビューダイジェスト」**と**「🧪人力チェック手順」**を読む。冒頭に**カード**を出す:

```
━━━━━━━━━ レビュー ① / N ━━━━━━━━━
🧪 [リスク] タスク名  （エピック）
変更概要: （📋ダイジェストの1〜2行）
本番反映に要るもの: （📋の env前提 / migration / functions deploy）
⚠️ 要注意: （📋の Gemini二重レビュー指摘・⚠️要注意 があれば必ずここに赤字で。無ければ「なし」）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━
🔗 テスト先（dev未push＝ローカルで確認）
  - admin: http://localhost:3001<該当パス>
  - liff : http://localhost:3000<該当パス>
  - DB   : ローカルSupabase（http://127.0.0.1:54321 / test アカウント）
  - 使うデータ/前提: （🧪手順の「テストデータ・前提」をそのまま）
```

- **URLは必ずフルで具体化**する。🧪手順が画面名やパス（例: admin `/車両`→`/vehicles`、`/non-submitters`、liff `/checkin/<siteId>`）を指していれば、`http://localhost:3001` または `:3000` を前置した**クリックできる完全URL**にして出す。siteId 等の可変部はローカルにテスト用データを用意（または既存IDを案内）して**実URLを埋める**。
- 🔴高・🧱土台で ⚠️要注意（Gemini指摘等）がある場合は、**「この指摘を承知でレビューしますか？」を最初に確認**してから手順に入る。

### リスク別の検証手数（人の手数をリスクで変える）
- **🟢低**：本番待ち直行のためここに来ない（ノールック）。
- **🟡中**：要点ステップのみ。ただし**「保存→再読込で永続するか」は必ず1ステップ入れる**（UI追加だけの見落とし＝DB制約/保存漏れを人の目でも拾う。例: 区分/enum追加）。
- **🔴高/🧱土台**：詳細ステップ＋**CCが事前にデモデータを用意して"実URLで差が見える"状態にしてから提示**（人にデータ作成させない）。金額/集計系は「変更前後で数値が分かれる」を必ず目視させる。Gemini指摘は最初に⚠提示し承知確認。
- **デモデータの罠（今回の教訓）**：CCがデモを用意する時は ①insert失敗を verify する ②**E2Eと同じ user×日付で衝突しない**（daily_reports は user×date 一意・upsert/afterAll で巻き込む）③検証後 cleanup。「実機が一致しない」時はまず**ブラウザのハードリロード＋dev server最新性＋データの実在**を切り分ける（コード結論を急がない）。

## §2 ステップ・ナビ（🧪手順の「👁 人間が見る点だけ」を1ステップずつ）

🧪手順の「👁 人間が見る点だけ」の番号付きステップを、**1つずつ**この形で提示し、人間の応答を待つ:

```
ステップ k/m
📍 場所: <完全URL>
🖱 操作: <何をする>
✅ 期待: <こうなれば合格>
（⚠ Gemini指摘の該当があればここに「特にここを見て」を併記）

→ 「OK」/「NG（内容）」/「skip」/「中断」 で答えてください
```

- 「✅ 自動で確認済み」項目は**再確認させない**（最初に「E2E/build/typecheck/RLSで担保済み＝省略」と1行で示すだけ）。人間には自動で拾えない点（見た目/文言/UX/実外部配信の体裁/主観ロジック）だけを見せる。
- **実外部副作用（メール/LINEの実送信）を伴うステップは、必ず「自分宛・自テナント・隔離」で行うよう明示**し、確認後 cleanup を促す。他人・全体配信は絶対にさせない。レビュー段階(dev)では原則 dry-run / 自分宛に限定。
- 人間が「NG」と言ったら、その内容を**そのステップに紐づけて記録**（後述§3で本文へ）。残りのステップは続行 or 中断を人間に選ばせる。
- 1タスクのステップを全部終えたら§3へ。

---

## §3 判定とステータス反映

全ステップ終了後、結果を集計して人間に最終確認:

### A. 全部OK（指摘なし）＝レビュー承認
- ステータスを **レビュー待ち → 本番待ち** に反映。**必ず REST PATCH で更新し、反映を再確認**する（MCPの`update_properties`はstatus取りこぼし実績あり）:
  ```bash
  node -e '/* .env の NOTION_TOKEN で */ fetch("https://api.notion.com/v1/pages/<id>",{method:"PATCH",headers:{Authorization:`Bearer ${TOKEN}`,"Notion-Version":"2025-09-03","Content-Type":"application/json"},body:JSON.stringify({properties:{"ステータス":{status:{name:"本番待ち"}}}})})'
  ```
  （実体は対話中にCCが組み立てて実行。更新後 GET で `本番待ち` を確認。）
- 本文に **`> [!note] ✅ 人力レビュー承認（CC nav / 日付）: 全〇ステップOK。reviewer=人間。本番待ちへ。`** を追記。
- 1行サマリ（「①〈タスク〉→本番待ち（承認）」）を出して次のタスクへ。

### B. 指摘あり（NGステップが1つ以上）
- 本文に **`> [!warning] 🔴 レビュー指摘（CC nav / 日付）`** callout を追記し、**NGステップ・期待・実際・場所URL**を箇条書きで残す。
- 人間に選ばせる:
  - **(1) 今すぐ直す** → CCがそのまま修正に入る（featureブランチ特定 or dev上で修正→ゲート緑→devへ→再度この§2から）。
  - **(2) 後で /run に直させる** → ステータスを **レビュー待ち → 進行中** に戻す（REST）。`🤖AI実行中`タグは付けない（人手再開ではないが、/runのstep1が痕跡=featureブランチ/指摘calloutで再開できる）。指摘calloutが直し先。
- どちらでも本番待ちには上げない。1行サマリ（「①〈タスク〉→指摘N件（(1)修正中/(2)進行中へ差戻し）」）。

### C. 中断
- そのタスクは**レビュー待ちのまま**（途中までのOK/NGは本文にメモ）。次タスクに行くか終了するかを人間に確認。

---

## §4 次のタスク → 枯渇 or 中断まで繰り返し

- §1〜§3 を**レビュー待ちが尽きるまで**優先度順に繰り返す（1件終えるごとに次を提示。人間が「終了」と言えば止める）。
- 全件終わったら**総括**: 承認→本番待ちにした件数／指摘で差し戻した件数／残レビュー待ち。
- **本番待ちが溜まったら案内**: 「本番反映は /ship（エピック単位でまとめ可・🔴高/🧱土台は本番スモークしてから・🟢はノールック可）」。**/review は ship をしない**。

## 厳守
- **push / deploy / db push / 本番migration適用 / ship はしない**（本番反映は /ship 専用・人間ゲート）。
- ステータス更新（レビュー待ち→本番待ち / →進行中）は **REST PATCH で実行し、更新後に必ず再確認**（MCP update_properties は status を取りこぼす実績あり）。
- **人間がOKと言ったステップだけ**合格にする。CCが勝手に「OK」にしない。自動で担保済みの項目は省略を明示。
- **実外部送信（メール/LINE）は自分宛・自テナント・隔離のみ**。他人/全体配信は絶対にしない。dev段階は dry-run 優先。
- ⚠️要注意（Gemini指摘等）は**隠さず最初に提示**し、人間が承知の上で承認するかを確認する。
- 対象は自案件（`.env` の `BACKLOG_PROJECT_ID`）のレビュー待ちのみ。
