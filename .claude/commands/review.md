# /review — レビュー待ちを優先順位順に人力テストをステップ・ナビ → 承認で本番待ちへ

> **【正本（~/cc-pipeline）・プロジェクト中立版】** 固有値をハードコードしない正本。**対象アプリ構成（apps/web 単体／web+mobile／web+LIFF 等）・画面パス・ローカルポート・本番ref・外部送信媒体（LINE/メール/push 等）は各リポの CLAUDE.md「Pipeline設定」の `APP_LAYOUT` / `APP_LAYOUT_NOTES` を参照**して具体化する。`{{DEV_URL}}` / `{{DEPLOY_PLATFORM}}` 等の `{{...}}` は CLAUDE.md「Pipeline設定」/`.env` 由来。UI/ロジックは原則ブラウザ（{{DEV_URL}}）で確認し、ネイティブ/LINEアプリ内など**実機依存は `⚠実機確認` として記録**する。

`/run` がレビュー待ちに積んだタスクを、**人間が１件ずつ手で確認するための対話ナビ**。各タスク本文の「🧪人力チェック手順」を**1ステップずつ提示**し、**毎ステップにテスト用URL・テナント/アカウント・期待結果を明示**して、人間の「OK/NG」を受けて進める。全ステップOKなら**レビュー承認＝ステータスをレビュー待ち→本番待ちに反映**する（CLAUDE.mdの唯一の人間ゲート「レビュー待ち→本番待ち昇格」を、この対話を通じて人間が実行する）。

> **大前提**: レビュー待ちのコードは **dev にマージ済みだが未push＝本番には出ていない**（{{DEPLOY_PLATFORM}}・dev はデプロイ対象外）。だから人力テストは原則 **ローカルで dev ブランチを動かして**行う。本番URLでの確認は ship 後（/ship）。本コマンドは「本番に出す前の最終人間チェック」。
> 承認後の本番反映は **/ship**（別の人間ゲート）。本コマンドは push / deploy / ship をしない。

引数 `$ARGUMENTS`:
- 空 … 自案件のレビュー待ちを**優先順位順**（緊急>高>中>低、同率は作成日古い順）に上から順にナビ。
- NotionページURL/ID/タスク名 … その1件だけをナビ。
- `--list` … レビュー待ち一覧（優先度順・リスク・エピック）を出すだけで終了（ナビしない）。

---

## §0 プリフライト（1回）

1. **対象取得**: `node --env-file=.env scripts/next-target.mjs --board` を実行し、**「レビュー待ち」グループ**を読む（既に優先度順＋作成古い順にソート済み・各行は `[優先順位/リスク🧱土台]` 形式）。`$ARGUMENTS` 指定時は notion-fetch で直接取得。0件なら「レビュー待ちなし」を報告して終了。
2. **ローカル環境を起動**（dev コードをローカルで触るため・具体値は CLAUDE.md「Pipeline設定」LOCAL_STACK / APP_LAYOUT_NOTES）:
   - Supabase ローカル: 落ちていれば `supabase start`（ポートは LOCAL_STACK の設定に従う）。
   - **web（主）**: {{DEV_URL}}（未起動なら `pnpm dev` 等＝APP_LAYOUT の起動コマンド）。管理画面・顧客導線とも**原則ここ＝ブラウザで確認**する。
   - **local接続assert**（本番を誤って触らない）: 配信される Supabase URL が **local（LOCAL_STACK のポート）**を指し、**本番 ref（CLAUDE.md「Pipeline設定」記載の本番Supabase ref）ではない**ことを確認（`curl -s {{DEV_URL}}/ | grep -oE '<localポート>|<本番ref>'`）。本番 ref を検出したら停止し `.env.local` で local 起動し直す。
   - **ブランチ確認**: レビュー対象のコードが今のローカルに居るか。基本 `dev` を見る（`git branch --show-current` が dev か、無ければ `git checkout dev`）。作業中の差分があれば退避を促す。
3. **一覧プレビュー**: レビュー待ちを `① [高/🔴] タスク名（エピック）` の形で番号付き提示。`--list` ならここで終了。

## §1 各タスクのナビ開始（上から1件ずつ）

対象ページを notion-fetch し、本文の **「📋レビューダイジェスト」**と**「🧪人力チェック手順」**を読む。冒頭に**カード**を出す:

```
━━━━━━━━━ レビュー ① / N ━━━━━━━━━
🧪 [リスク] タスク名  （エピック）
変更概要: （📋ダイジェストの1〜2行）
本番反映に要るもの: （📋の env前提 / migration / cron・deploy）
⚠️ 要注意: （📋の Gemini二重レビュー指摘・⚠️要注意 があれば必ずここに赤字で。無ければ「なし」）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━
🔗 テスト先（dev未push＝ローカルで確認）
  - web   : {{DEV_URL}}<該当パス>
  - 実機  : ⚠ ネイティブ/LINEアプリ内など実機固有（APP_LAYOUT_NOTES の該当導線）は実機確認。ブラウザで足りる UI/ロジックは {{DEV_URL}} で確認
  - DB    : ローカル Supabase（LOCAL_STACK のポート / テスト テナント・アカウント）
  - 使うデータ/前提: （🧪手順の「テストデータ・前提」をそのまま）
```

- **URLは必ずフルで具体化**する。🧪手順が画面名やパス（※実パスはプロジェクト固有＝CLAUDE.md「Pipeline設定」APP_LAYOUT_NOTES の画面パス例に合わせる）を指していれば、`{{DEV_URL}}` を前置した**クリックできる完全URL**にして出す。可変部（各種ID）はローカルにテスト用データを用意（または既存IDを案内）して**実URLを埋める**。
- **テスト先URLは必ず Markdown のクリック可能リンク `[URL](URL)` 形式で出力する**（生のテキストURLにしない＝ターミナルでそのまま開けるように）。カードの「🔗 テスト先」も §2 ステップの「📍 場所」も同様。Notion等の参照リンクも `[ラベル](URL)` で出す。
- 🔴高・🧱土台で ⚠️要注意（Gemini指摘等）がある場合は、**「この指摘を承知でレビューしますか？」を最初に確認**してから手順に入る。

### リスク別の検証手数（人の手数をリスクで変える）
- **🟢低（純・要人力確認カテゴリ非該当）**：本番待ち直行のためここに来ない（ノールック）。
- **🟢低でも「要人力確認カテゴリ該当」でここに来た場合（2026-07-02〜）**：/run が **金額/集計/データ統合・マージ/migration/認証・RLS/外部送信** に触れると判定してレビュー待ちへ回した🟢低。リスクは低くても**ミスの影響が大きい**ので、要点ステップ＋**「統合/集計/金額の結果が正しいか（統合漏れ・二重計上・孤児が無いか）を目視」**を必ず入れる（例：現場マージ後に現場別集計が統合先へ合算されているか）。
  - **★影響範囲マップの全参照を確認する（breadth）**：📋 の「依存/影響範囲」に /run が洗い出した参照（他テーブル・JSON内ID・各画面・集計/PDF/エクスポート・EF/バッチ）が並んでいる。**主画面だけでなく、その各参照が変更後も正しいかを1つずつ確認**する。今回の現場マージ事故は「マージ操作成功」だけ見て**現場別集計（下流の参照）を見落とした**＝影響範囲の広さを見る観点が主眼。📋に影響範囲マップが無い/薄いと感じたらそれ自体を指摘（=範囲判断不足）。
- **🟡中**：要点ステップのみ。ただし**「保存→再読込で永続するか」は必ず1ステップ入れる**（UI追加だけの見落とし＝DB制約/保存漏れを人の目でも拾う）。
- **🔴高/🧱土台**：詳細ステップ＋**CCが事前にデモデータを用意して"実URLで差が見える"状態にしてから提示**（人にデータ作成させない）。金額/集計系は「変更前後で数値が分かれる」を必ず目視させる。Gemini指摘は最初に⚠提示し承知確認。
- **デモデータの罠**：CCがデモを用意する時は ①insert失敗を verify する ②**対象テーブルの一意制約・upsert/on conflict 衝突に注意**（複合一意・PK 構成はプロジェクトのデータモデル参照）。**RLS スコープ（テナント/組織/所有者＝CLAUDE.md「Pipeline設定」RLS構成メモ）に合わせてシード**しないと自分のセッションから見えない ③検証後 cleanup。「実機が一致しない」時はまず**ブラウザのハードリロード＋dev server最新性＋データの実在（＋RLSスコープ）**を切り分ける（コード結論を急がない）。

### 🖥 app固有の確認方針（APP_LAYOUT_NOTES 準拠）
対象アプリの UI/ロジックは**原則ブラウザ（{{DEV_URL}}）で確認する**。**ネイティブ依存・LINEアプリ内固有（プッシュ通知の実機体裁・カメラ・APNS/FCM・Flexメッセージ体裁・友だち追加導線・課金の実機フロー 等＝APP_LAYOUT_NOTES の該当項目）**は**『⚠実機確認が必要』**とステップに注記する。**現段階はブラウザ確認で承認可**、実機固有は**NGにせず⚠保留として記録**（§3-B の指摘ではなく「⚠実機確認 残」として本文にメモし、ブラウザで確認できた範囲で承認してよい）。
**外部送信（LINE配信/メール/push 等＝APP_LAYOUT_NOTES の送信媒体）を伴う確認は必ず「自テナント・自分宛・隔離・dry-run優先」**で行い、確認後 cleanup する。他テナント・実顧客への実配信は絶対にしない。

## §2 ステップ・ナビ（🧪手順の「👁 人間が見る点だけ」を1ステップずつ）

🧪手順の「👁 人間が見る点だけ」の番号付きステップを、**1つずつ**この形で提示し、人間の応答を待つ:

```
ステップ k/m
📍 場所: [<完全URL>](<完全URL>)   ← 必ずクリック可能な Markdown リンクで出す
🖱 操作: <何をする>
✅ 期待: <こうなれば合格>
（⚠ Gemini指摘の該当があればここに「特にここを見て」を併記）

→ 「OK」/「NG（内容）」/「skip」/「中断」 で答えてください
```

- 「✅ 自動で確認済み」項目は**再確認させない**（最初に「E2E/build/typecheck/RLS監査で担保済み＝省略」と1行で示すだけ）。人間には自動で拾えない点（見た目/文言/UX/外部送信の体裁/主観的な業務ロジック）だけを見せる。
- **実外部副作用（LINE配信/メール/push 等）を伴うステップは、必ず「自テナント・自分宛・隔離」で行うよう明示**し、確認後 cleanup を促す。他テナント・実顧客への実配信は絶対にさせない。レビュー段階(dev)では原則 dry-run / 自分宛に限定。
- 人間が「NG」と言ったら、その内容を**そのステップに紐づけて記録**（後述§3で本文へ）。残りのステップは続行 or 中断を人間に選ばせる。
- 1タスクのステップを全部終えたら§3へ。

---

## §3 判定とステータス反映

全ステップ終了後、結果を集計して人間に最終確認:

### A. 全部OK（指摘なし）＝レビュー承認
- ステータスを **レビュー待ち → 本番待ち** に反映。**必ず REST PATCH で更新し、反映を再確認**する（MCPの`update_properties`はstatus取りこぼし実績あり）:
  ```bash
  node -e '/* .env の NOTION_TOKEN で */ fetch("https://api.notion.com/v1/pages/<id>",{method:"PATCH",headers:{Authorization:`Bearer ${TOKEN}`,"Notion-Version":"2025-09-03","Content-Type":"application/json"},body:JSON.stringify({properties:{"ステータス":{status:{name:"本番待ち"}}}})})'
  ```
  （実体は対話中にCCが組み立てて実行。更新後 GET で `本番待ち` を確認。）
- 本文に **`> [!note] ✅ 人力レビュー承認（CC nav / 日付）: 全〇ステップOK。reviewer=人間。本番待ちへ。`** を追記。
- 1行サマリ（「①〈タスク〉→本番待ち（承認）」）を出して次のタスクへ。

### B. 指摘あり（NGステップが1つ以上）
- 本文に **`> [!warning] 🔴 レビュー指摘（CC nav / 日付）`** callout を追記し、**NGステップ・期待・実際・場所URL**を箇条書きで残す。
- 人間に選ばせる:
  - **(1) 今すぐ直す** → CCがそのまま修正に入る（featureブランチ特定 or dev上で修正→ゲート緑→devへ→再度この§2から）。
  - **(2) 後で /run に直させる** → ステータスを **レビュー待ち → 進行中** に戻す（REST）。`🤖AI実行中`タグは付けない（人手再開ではないが、/runのstep1が痕跡=featureブランチ/指摘calloutで再開できる）。指摘calloutが直し先。
- どちらでも本番待ちには上げない。1行サマリ（「①〈タスク〉→指摘N件（(1)修正中/(2)進行中へ差戻し）」）。

### C. 中断
- そのタスクは**レビュー待ちのまま**（途中までのOK/NGは本文にメモ）。次タスクに行くか終了するかを人間に確認。

## §4 次のタスク → 枯渇 or 中断まで繰り返し

- §1〜§3 を**レビュー待ちが尽きるまで**優先度順に繰り返す（1件終えるごとに次を提示。人間が「終了」と言えば止める）。
- 全件終わったら**総括**: 承認→本番待ちにした件数／指摘で差し戻した件数／残レビュー待ち。
- **本番待ちが溜まったら案内**: 「本番反映は /ship（エピック単位でまとめ可・🔴高/🧱土台は本番スモークしてから・🟢はノールック可）」。**/review は ship をしない**。

## 厳守
- **push / deploy / db push / 本番migration適用 / ship はしない**（本番反映は /ship 専用・人間ゲート）。
- ステータス更新（レビュー待ち→本番待ち / →進行中）は **REST PATCH で実行し、更新後に必ず再確認**（MCP update_properties は status を取りこぼす実績あり）。
- **人間がOKと言ったステップだけ**合格にする。CCが勝手に「OK」にしない。自動で担保済みの項目は省略を明示。
- **実外部送信（LINE配信/メール/push 等）は自テナント・自分宛・隔離のみ**。他テナント/実顧客への実配信は絶対にしない。dev 段階は dry-run 優先。
- ⚠️要注意（Gemini指摘等）は**隠さず最初に提示**し、人間が承知の上で承認するかを確認する。
- 対象は自案件（`.env` の `BACKLOG_PROJECT_ID`）のレビュー待ちのみ。
