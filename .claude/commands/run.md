# /run — 優先順位明確・止まらない自走ループ

CLAUDE.md「自走ポリシー」に従い、**バックログ（`.env` の `BACKLOG_PROJECT_ID` で絞った自案件）を唯一の真実**として、毎ループ最優先の1件を選び **実装まで自走** する単一ループ。技術・手続き的判断は既定ポリシーで自分で決める。**人間の業務判断が要るものだけ「要回答」にパークし、ループは止めず次へ**進む。**唯一の人間ゲートは本番反映（push / ship）だけ**。

> 旧 `--advance` / `AUTO_ADVANCE` は廃止。`/run` は既定で要件化だけにとどまらず実装まで進む。

引数: `$ARGUMENTS`
- `--dry-run` … 盤面と「各件をどのフェーズに動かすか」を提示するのみ（Notion更新・コード変更・通知・ブランチ・migration適用を一切しない）。副作用ゼロ。

**停止モデル＝キュー枯渇まで回り切る**（下記「終了条件」）。`MAX_ITEMS` / `MAX_MINUTES` は**通常の停止条件ではない**（残予算を理由に大型アイテムの着手を見送らない）。暴走・課金の**回路遮断**として `MAX_WALL=180分` と「連続前進ゼロ K=3」だけを持つ。

**人ボールを出す時は CLAUDE.md「人ボール報告の必須フォーマット（結論ファースト）」に従う**：画面は「🙋 [種別] タスク名／【決めてほしいこと】／【選択肢】A/B/C／【推奨】X」。`--detail` は1〜2行に圧縮。

---

## 0. 開始時の自己修復（ループ前に1回）
自案件のタスクで **「🤖AI実行中」タグが付いたまま残っているもの**があれば**全部外す**（前回クラッシュの取り残しクリア）。`--dry-run` 時は「外す対象」を列挙のみ。

## 1. 盤面取得（全ステータス）
`node scripts/next-target.mjs --board` で自案件の全ステータススナップショットを取得（案件で絞り済み）。各タスクの ステータス / 優先順位 / タグ / URL が得られる。

---

## 優先順位（毎ループこの順で対象を1件選ぶ）

| 順 | 区分 | /run の動作 | 遷移先 |
|---|---|---|---|
| **0** | 🤖タグ掃除 | 取り残しの「🤖AI実行中」を外す（1回） | — |
| **1** | **進行中**（再開可能なCC痕跡あり） | 仕掛りを先に終わらせる：設計→実装→検証→devマージ→レビュー待ち。痕跡が無い＝人手の進行中は触らない | レビュー待ち |
| **2** | **未整理**（棚卸し・速い） | 各件を判定：**(a)** 業務判断が必要 → **要回答**化＋通知、止まらず次へ／**(b)** 要件化可能 → 薄い要件を書いて **要件定義済み**＋**優先度タグ（緊急/高/中/低）**付与 | 要回答 / 要件定義済み |
| **3** | **要件定義済み**（実装） | 優先度 **緊急>高>中>低**、同率は作成古い順で1件実装：着手→実装→検証→devマージ→**レビュー待ち**。上限まで連続で回す | レビュー待ち |
| — | 要回答 / レビュー待ち / 本番待ち / 保留 / 完了 | 触らない・再通知しない（ゲート/終端） | 不変 |

**着手＝Notionを「進行中」に更新＋「🤖AI実行中」タグ付与（既存タグ保持して追加）。レビュー待ちへ上げる時／要回答パーク時／中断時に必ず外す**（付けっぱなし厳禁・原則同時1件）。

**未整理の棚卸しを先に回し切る**（step2）ことで、新規に入った緊急/高が step3 の実装で正しい優先度順に拾われる。「未着手だから検知できない」状態を作らない。

---

## 2. メインループ（擬似コード）

```
board = next-target.mjs --board
processed = ∅                    # 1ループ内で同一 page_id を二度処理しない
parked    = []                   # この実行で要回答にパークした業務判断（末尾に提示）
gated     = []                   # この実行でレビュー待ちに積んだもの（通知対象）
t0 = now

selfHeal()                       # step0: 🤖タグ掃除

# step1: 再開可能な進行中を先に片付ける
for item in 進行中(痕跡あり):
    if elapsed() >= MAX_WALL(180min): break          # 回路遮断のみ
    実装を完了 → 検証(必要ならローカルstack起動) → devマージ
    if 検証が N=2回 green にできない: set 保留; 記録; blocked += item; continue
    set レビュー待ち; タグ外す; gated += item

# step2: 未整理を全件棚卸し（速い・実装しない）
for item in 未整理(優先度→作成順):
    if 業務判断が必要(a): set 要回答; notify(要回答); parked += item; continue
    else (b):            薄い要件＋AC＋scope を書く; 優先度タグ付与; set 要件定義済み

# step3: 要件定義済みを優先度順に実装（キュー枯渇まで・上限なし）
noProgress = 0
while 実装可能な要件定義済みがある():               # = 未処理かつ非blockedの要件定義済み
    if elapsed() >= MAX_WALL(180min) or noProgress >= 3: break   # 回路遮断（異常時のみ）
    item = 最優先(緊急>高>中>低, 同率は作成古い順, processed/blocked除外)
    set 進行中 + 🤖タグ
    設計（既存パターン準拠・決定理由を1行記録）→ 実装 → 検証 → devマージ
    # ── per-item ガード（これが無いと枯れない／暴走する）──
    if 途中で業務判断が必要と判明:                 # 人間の業務判断
        set 要回答(A/B/C+推奨); タグ外す; notify; parked += item; processed.add(item.id); noProgress=0; continue
    if 検証が N=2回 green にできない:               # サイズではなく失敗
        set 保留; 失敗内容をタスクに記録; タグ外す; blocked += item; processed.add(item.id); noProgress += 1; continue
    # サイズが大きいだけなら止めず実装する（予算を理由に見送らない）
    writeReviewDigest(item)                      # 📋レビューダイジェスト追記＋リスク/🧱土台タグ（下記必須）
    set レビュー待ち; 🤖タグ外す; gated += item; processed.add(item.id); noProgress = 0

# ループ後
for item in gated:  notify(レビュー待ち, item)   # 到達時1通（既存パーク分は通知しない）
report(盤面 / 実装したもの / parked まとめ / 本番ゲート待ち)
```

## 📋 レビューダイジェスト（レビュー待ちへ移す時＝必須）
バッチレビューを成立させるため、アイテムを**レビュー待ちに移す時**は当該ページに**必ず**以下を構造化して書く（既存のreviewer noteは置き換える）。

### 1. ページ本文に「📋レビューダイジェスト」callout を追記
- **変更概要**: 1〜2行（何を・どこに）
- **リスク**: 高 / 中 / 低
  - **高** = schema/migration・共有util・データモデル・認証/RLS・課金・外部送信 に触れる
  - **中** = 複数画面/複数ファイルにまたがる機能追加
  - **低** = 自己完結のUI/文言/単一画面のトグル等
- **依存**:
  - 「土台」: この変更に乗る後続アイテム（未着手/要件定義済み）があれば 件名/page_id を列挙
  - 「乗っかり先行」: この変更が依存する既マージの先行アイテム
- **手動確認ポイント**: E2Eで拾えない所だけ（英語表現・UX・通知本文・金額表示 等）
- **推奨shipグループ**: 所属エピック（例「下請け取引管理」「現場管理」「i18n」）

### 2. リスクをタグでも付ける（レビューキューのソート用）
- **🔴高リスク / 🟡中リスク / 🟢低リスク** のいずれかを付与（既存タグは保持）。
- 「他の未着手/要件定義済みが依存する土台」と判定したら **🧱土台** も付与。

### 3. 土台アイテムは“都度レビュー推奨”を明示
- **🧱土台** に該当する場合、レビュー待ち通知＆run末尾サマリに
  「⚠️これは土台（後続N件が依存）。バッチに溜めず早めにレビュー推奨」を1行入れる。

> 狙い: 人間は「🔴と🧱だけ実物spot check」「🟢leafはバッチで溜めてOK」で30件でも捌ける。

## 自走ポリシー（ここでは聞かない・自分で決めて記録する）
- **検証環境**: AC が E2E/DB を要求するなら、ローカルスタック（Docker / `supabase start`）を**自分で起動**して回す。落ちていたら起動する。聞かない。
- **🔒 プリフライト必須（E2E/検証の前に毎回）**: 起動中の dev/stack が**必ず local 接続か**を assert する（配信 anon key == local の `sb_publishable…` を確認。`eyJ…`＝本番JWTを検出したら **検証を止める**）。本番key検出時は **local で再起動**してから回す。**本番に対してE2Eを絶対に走らせない**（本番データ汚染・破壊防止）。例: `curl -s localhost:3000/ | grep -oE 'sb_publishable_[A-Za-z0-9]{6}|eyJ[A-Za-z0-9]{6}'` がcloudなら停止→再起動。
- **技術選定・ライブラリ・スコープ**: 既存パターン（近い画面/関数/テスト）に合わせて**自分で決定**し、タスクに「**決定と理由**」を1行記録して進む。
- **要件の曖昧さ**: 設計メモ・既存コードから意図が**一意に取れる**なら (b) として確定し進む。
- **migration**: 追加のみDDL（ADD COLUMN / CREATE TABLE / CREATE INDEX 等）は**作成＋ローカル適用**してよい（`db reset` 等）。**本番適用はしない**（人間ゲート）。
- 上記でも“**一意に決まらない業務判断**”（UX方針・事業判断・テナント/LINE構成・通知先 等）**だけ** 要回答（A/B/C＋推奨）。パークしたら**次の対象へ。run 全体は止めない**。

## 人間ゲートと「パーク＆継続」
- **唯一の“停止”ゲートは本番反映（remote push / 本番デプロイ / ship）**＝人間の明示OKまで実行しない。
- それ以外（実装・ローカル検証・migrationのローカル適用・devマージ・レビュー待ち移動）は**自走でOK**。
- **dev は未push・本番影響なし前提で自由に積む**（`vercel.json` で dev デプロイ無効）。
- **🅿️ 人間判断アクションは「パーク＆継続」**: 本番削除・業務判断・本番反映など人間の確認が要る**“そのアクションだけ”を保留**にし、**run は止めない**。残りの実装対象を**先に全部進め**、run 末尾に**まとめて提示**（1問1答・A/B/C＋推奨）。
  - **例外（待機してよいのは依存時のみ）**: 残りの実装対象が**その回答に依存して進められない**場合だけ、その対象をブロック扱いにして待つ（他に対象が無ければ停止＝下記「依存ブロック」）。

## 終了条件（キュー枯渇まで・任意中断や予算切りは禁止）

### 主停止条件＝アクション可能な対象の枯渇（正常終了）
- **未整理=0 かつ 要件定義済みに「実装可能な残り」が無い**（全件が レビュー待ち / 本番待ち / 要回答 / blocked(保留) / 完了 に収まった）まで**回り切る**。
- **レビュー待ち・本番待ちは何件溜まってもOK**。停止理由にしない。
- **MAX_ITEMS / MAX_MINUTES は停止理由にしない**。残予算を理由に大型アイテムの着手を見送らない（サイズが大きいだけなら止めず実装する）。

### per-item ガード（枯渇させるための前進保証・run全体は止めない）
- **人間の業務判断が要る** → 要回答 にパーク（A/B/C＋推奨・通知）→ 次へ。
- **検証が N=2 回 green にできない** → **blocked**（ステータス=保留＋失敗内容をタスクに記録・🤖タグ外す）→ 次へ。
- **サイズが大きいだけ** → 止めず実装する。

### バックストップ（回路遮断・異常時のみ。通常停止ではない）
- **直近 K=3 件連続で前進ゼロ**（＝3連続 blocked 等で1件もレビュー待ちに積めない）→ 安全停止。
- **壁時計 MAX_WALL=180分 超過** → 安全停止。
- これらは暴走・課金防止のためだけ。正常時は主停止条件で終わる。

### 停止時サマリ（必須）
- **主停止（枯渇）か バックストップ（異常）か**を明示。
- **レビュー待ち / 要回答パーク / blocked(保留)** の**件数を一覧**で。
- まだ実装可能な残りがあるのにバックストップで止めた場合のみ「**再 `/run` で続き**」を明記。

## 冪等・安全
- 各フェーズは**前段が完了したタスクだけ**前進（多重発火しない）。1ループで同一 page_id を二度処理しない。
- ゲート/終端（要回答・レビュー待ち・本番待ち・保留・完了）は触らない・再通知しない。
- 自プロジェクト（`BACKLOG_PROJECT_ID`）の page_id 以外は対象外（`--board` が案件で絞済み）。
- **破壊的操作（ファイル削除等）は実行前に確認**。自分の生成物でない残骸（macOSの「 2」複製等）は**触らない**。

## 3. 通知（フロー2：既存 LINE human-in-the-loop に1通）
**この実行でそのゲートに“入れた”時だけ1通**（既存パーク分は再通知しない＝多重発火防止）。本番ゲートの文面は PRマージ承認と**混線させない**。
```bash
# 要回答（未整理→要回答 / 実装中に業務判断が必要と判明）
node scripts/notify-humanball.mjs --kind 要回答     --task "<タスク名>" --detail "<決めてほしいこと＋A/B/C＋推奨を1〜2行>" [--url "<セッションurl>"]
# レビュー待ち（実装完了・devマージ済み）
node scripts/notify-humanball.mjs --kind レビュー待ち --task "<タスク名>" --detail "実装完了・devマージ済み。レビューOKなら本番待ちへ。<PR/ブランチ>" [--url "<セッションurl>"]
# 本番ゲート（push / ship 待ち）— /ship の ship承認とは別文面・別kindで
node scripts/notify-humanball.mjs --kind 要対応     --task "<タスク名>" --detail "本番反映待ち。push/ship の明示OKを" [--url "<セッションurl>"]
```
- 1停止で複数承認をまとめる場合のみ通知1回（連投しない）。失敗しても best-effort で続行。

## 4. 出力（/run 完了サマリ）
- **やった実装**（1行 × 件数）
- **盤面**（各ステータス件数）
- **レビュー待ちに積んだもの**（各件に **🔴/🟡/🟢** と **🧱土台** を併記。**🧱土台は冒頭で「早めにレビュー推奨」と強調**）
- **blocked(保留)**（検証N=2回green不可・失敗内容つき）の件数一覧
- **要回答にパークした業務判断**（まとめて・1問1答・A/B/C＋推奨）
- **本番ゲートで待っているもの**（push / ship 待ち）
- 主停止（枯渇）か バックストップ（異常）かを明示。バックストップ時は「再 `/run` で続き」を明記。

## 厳守
- 本番反映（push / 本番デプロイ / db push / ship）は人間の明示OKまで**実行しない**。それ以外は自走。
- 人ボール化（要回答）したタスクは人の回答が入るまで再着手しない（同じ問いでループしない）。
- 「🤖AI実行中」タグは“CCが実装中”の印。着手で付け、レビュー待ち/要回答/中断で必ず外す。原則同時1件。
- `--dry-run` は副作用ゼロ。いつでも中断可能。
