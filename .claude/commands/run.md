## ステータス遷移（AUTO_MERGE_TARGET=dev）

```
未整理 ──振り分け──▶ 要件定義済み / 要回答(業務判断)
要件定義済み ─（優先順位順に実装）+テスト全緑─┐
   ├─【低リスクレーン】(リスク ≤ AUTO_TIER＝🟢低 かつ 要人力確認カテゴリ非該当) ─▶ dev自動マージ ─▶ 本番待ち（レビュー不要・ship待ち）
   └─【中/高/土台レーン】(🟡🔴🧱 または 🟢低でも要人力確認カテゴリ該当) ─[🔴高・🧱土台は Gemini二重レビュー]─▶ レビュー待ち ─(人間レビュー＝人力テスト)─▶ 本番待ち
本番待ち（無レビュー低リスク＋レビュー済み中高が混在）─人間がエピック単位でまとめてship─▶ 完了
   （🟢低リスク分はノールック／🔴高・🧱土台分は本番レビューしてからship）
```
- **緊急**（`優先順位=緊急`）はリスク不問で**最優先割り込み**。**ただしマージ経路は dev ではなく「main派生（origin/{{PROD_BRANCH}}基点）の hotfix ブランチ」**で実装し（＝dev堆積から隔離して独立shipできるようにする）、ゲート緑まで自走 → リスク不問で **本番待ち** に置き **🚨緊急通知**。push/PR/Merge は人（/ship）。緊急は速さ最優先のため、🔴高でも レビュー待ち を経由せず本番待ち直行（ただし Gemini/RLS等のゲートと🧪人力チェック手順は通常どおり実施＝品質は落とさない／本番レビューは /ship 時に人が行う）。詳細は §緊急レーン。
- ※将来 `AUTO_MERGE_TARGET=prod` 切替時のみ：低リスクは本番待ちを経ず **完了** まで自動（現行 dev では必ず本番待ちで一旦停止＝本番影響ゼロ）。

## /run 統合フロー（自 page_id 内・枯渇まで自走）

### 0. プリフライト＋自己修復＋緊急監視（ループ前に1回＋毎周回の冒頭）
- **自己修復**: 自案件で「🤖AI実行中」タグが残っているものを全部外す（前回クラッシュの取り残し）。`--dry-run` は列挙のみ。
- **local接続 assert（本番E2E禁止）**: **{{LOCAL_STACK}}=none のプロジェクトはローカルスタックが無いため、この local接続 assert 自体をスキップ**（E2E/DB検証も実体が無ければ走らせない）。ローカルスタックがある場合のみ：E2E/DB検証を使う前に**毎回**、起動中の dev/stack が **local 接続**かを確認（配信される Supabase URL / anon key が **local（{{DEV_URL}} / 127.0.0.1:54321 等）**を指し、**本番プロジェクトの URL / JWT ではない**こと）。本番接続を検出したら**検証を止め**、`.env.local` で local 起動/再起動してから回す。**本番に対して E2E を絶対に走らせない**。例: `curl -s {{DEV_URL}}/ | grep -oE '127\.0\.0\.1:54321|<本番SUPABASE_URLの一部>'` で local 側に一致することを確認。Docker/`supabase start` が落ちていれば起動。聞かない。
- **緊急監視（常時）**: 毎周回の冒頭で `優先順位=緊急` の未着手（要件定義済み/未整理）が無いか見る。あれば§2でプリエンプト。
- **議事録の気づき（ループ前に1回）**: `{{MINUTES_DS_ID}}`（議事録テーブル）に **`処理状態=未処理` かつ `案件`=自案件（`BACKLOG_PROJECT_ID`）** の議事録があれば、**「未処理の議事録がN件あります。先に `/intake` で分解→未整理化するのを推奨」と1行案内**する（会議/電話由来の複数タスク源＝②パターン）。/run は**自動で分解しない**（分解は高判断＝抽出サマリを人が見られる /intake に任せる）。案内後は通常どおり要件定義済み/未整理の drain を続ける（議事録があること自体は停止理由にしない）。
### 1. 進行中（再開可能なCC痕跡あり）を先に完了
featureブランチ/設計書/🤖タグ のいずれかがある進行中を再開：実装→自動テストゲート(§5)→**当該リスクのレーンで着地**（低→§4／中高土台→§3）。痕跡が無い＝人手の進行中は触らない。

### 2. 未整理の棚卸し（統合・速い・実装しない）
各件を内容から判定し、**1件処理ごとに盤面先頭へ戻る**。判定時に以下を**まとめて自動付与**（リスク/土台は必ず該当 Select/Checkbox プロパティ値に入れる・本文/独自タグ禁止）：
- **【優先度】**（緊急/高/中/低）・**【エピック】**（shipグループ）・**【リスク】**（🔴高/🟡中/🟢低）・**【土台】**（後続が依存するか）。判定基準は §リスク・エピック自動判定。
- **(a) 業務判断が必要**（業務ルール/要件が曖昧・本文が空/不足）→ **要回答**化（A/B/C＋推奨）し**パーク**、止まらず次へ。
- **(b) 要件化可能** → 薄いストーリー＋AC＋scope を書き、上記4プロパティを付与して **要件定義済み**。

### 3〜4. 要件定義済みを実装（① 緊急を最優先割り込み → ② 残りは**優先順位順（緊急>高>中>低）** → 同順位は作成日古い順）
> **順序の主キーは「優先順位」**（緊急>高>中>低・同順位は作成日古い順）＝**重要なものから片付ける**。※旧仕様「リスク低い順」から **2026-07-02 に優先順位順へ変更**。リスク(🟢🟡🔴)は**順序に関与させない**＝リスクは §3/§4 の**レーン分岐**と**工数スケーリング**にのみ使う。
> - **① 緊急（`優先順位=緊急`）は特別扱い**：最優先割り込み（業務停止級）。**実装は dev ではなく main派生（`git fetch origin {{PROD_BRANCH}} && git switch -c hotfix/<slug> origin/{{PROD_BRANCH}}`）の hotfix ブランチ**で行い、ゲート緑まで自走 → **本番待ち**（レビュー待ちを経由しない）＋ **🚨緊急通知**。**push/PR/Merge はしない（人が /ship で行う＝本番ゲート）**。狙い＝マージ待ち(dev)がどれだけ膨らんでも、緊急だけは clean な main から分岐して**独立して即shipできる**。詳細手順は §緊急レーン。
> - **② それ以外は優先順位順（高→中→低）で実装**。同順位内は作成日古い順。優先度が高いものから着手し、そのタスクのリスクに応じて §3/§4 のレーンへ流す（下記 c.）。
> - **【厳守】優先順位を尊重し、着手前に勝手に順序を入れ替えない**。上位優先度が1件でも残る限り下位に先行しない。曖昧で着手できないものは §2(b) 要件化 or §2(a) 要回答で前進させる（黙って飛ばさない）。
> - **【厳守】1件だけ実装して止めない＝レビューはバッチで積む**。順に捌き、**できれば同じ `エピック` の案件をまとめて**レビュー待ちに積む（人が /review でエピック単位でまとめて確認できる状態にする）。レビュー待ちが溜まる事自体は停止理由ではない（§6①の枯渇 or §6③の異常まで自走）。
1件取り、`進行中`＋🤖タグにして実装：
- **a. 実装**：既存パターン（近い画面/関数/テスト）に準拠。タスクに「**決定と理由**」を1行記録。追加のみDDL（ADD COLUMN/CREATE TABLE/CREATE INDEX 等）は **作成＋local適用**（`db reset` 等）。**本番適用はしない**。実装中に業務判断が要ると判明したら**要回答**化してパーク・次へ。
- **b. 【自動テストゲート】今存在する自動テストを“全部”通す**（動的検出・§テストゲート）。緑でなければ **blocked**（ステータス=保留＋失敗内容を記録・🤖外す）→ 次へ（run 全体は止めない）。
- **c. レーン分岐**（`リスク` プロパティ ＋ **要人力確認カテゴリ**で判定）:

  > **★要人力確認カテゴリ（変更の性質で自動判定／リスク低でも人力テスト必須・2026-07-02追加）**：実装した**実差分**が次のどれかに触れるなら、**🟢低でも §4（自動マージ・レビュー不要）に流さず §3（レビュー待ち＝人力テストあり）へ回す**。狙い＝「見た目は軽いがミスの影響が大きい」変更を人の目で拾う（**実例：現場マージで現場別集計が統合されず見落とした**＝自動/人力テストの粒度不足）。
  > - **金額・原価・単価・料率** ／ **集計・按分・レポート/PDF出力の数値**
  > - **データ統合・マージ・付け替え・一括更新・移行**（複数レコードを1つにまとめる/移す系＝統合漏れ・孤児・二重計上が出やすい）
  > - **migration / スキーマ / データモデル**（追加のみでも保存・読み戻し経路に関わるもの）
  > - **認証・権限・RLS・anon露出・外部送信（LINE/メール/push/決済）**
  > 判定は**着手前の見立てではなく実装後の実差分**で行う。該当したら 📋 に「**要人力確認カテゴリ該当（理由）**」を1行残し、🧪人力チェック手順に「**統合/集計/金額の結果を目視**」を必ず1ステップ入れる。

  > **★影響範囲マップ（要人力確認カテゴリ＝特にデータ統合/マージ/データモデル/共有エンティティ変更は「着手時」に必須・2026-07-02追加）**：テスト範囲の前に**影響範囲（blast radius）を先に洗い出す**。今回の現場マージ事故の主因は「テスト不足」ではなく**影響範囲の判断不足**（現場IDを持つ他データ・どの画面が使うかを把握できていなかった）。手順：
  > 1. **対象エンティティ/ID を特定**（例: site の id）。
  > 2. **その参照を全コード＋DBスキーマ横断で grep/Explore**：①他テーブルのFK（`site_id` 等）②**JSON内に ID/名称で持つ箇所**（例 `daily_reports.sites`）③その値で絞り込む/表示する**画面**④**集計・按分・レポート・PDF・エクスポート**⑤**edge function/バッチ/cron**⑥キャッシュ/検索インデックス。
  > 3. **洗い出した各参照を「変更後も正しいか」のテスト観点に落とす**（統合/マージなら **全参照が統合先を向くか・マージ元が残って孤児/二重計上にならないか・集計が合算されるか**）。この観点を **(a)自動E2E (b)🧪人力チェック (c)ship前スモーク の3つ全部**でカバーする（下記）：
  >    - **(a) 自動E2E/verifyの範囲も広げる**：既存テストを流すだけで満足しない。**影響範囲マップの主要参照（特に下流の集計・関連・エクスポート）を検証するE2E/`verify-*.mjs`が無ければ1本追加する**（例：現場マージ→マージ先の現場別集計が合算＝マージ元の値が統合先に乗る、を assert）。＝「操作が成功した」でなく「統合結果が正しい」を自動で固定する。
  >    - **(b) 🧪人力チェック**：マップの各参照（画面/集計）を1ステップずつ目視（review.md がbreadthで確認）。
  >    - **(c) ship前スモーク**：§人力チェック手順の「🚀ship前スモーク」も**マップの各参照（影響先の画面・集計）を本番で確認**する範囲に広げる（主画面だけ見ない）。
  > 4. 既存の**消費箇所チェックリスト（`docs/*-consumers.md` があれば）を使う/無ければ作る/更新する**（例 [[project_expense_consumers]] の docs/expense-data-consumers.md と同型）。マップ結果は 📋 の「依存/影響範囲」に残す。
  > ※ この影響範囲マップは §工数スケーリングの「🔴高＝Exploreで全消費箇所をマップ」を、**リスク低でも要人力確認カテゴリなら必須**に広げたもの。

  **§4 低リスクレーン（`リスク` ≤ `AUTO_TIER`＝🟢低 **かつ 要人力確認カテゴリに非該当**）**
  - テスト全緑を確認 → `AUTO_MERGE_TARGET`（=dev）へ**自動マージ**（ローカルマージ・**push しない**）→ ステータス=**本番待ち**・🤖外す。**レビュー不要**（自己完結UI・文言・単一トグル・設定値など**ミスが波及しないもの限定**）。📋ダイジェストは軽め（変更概要＋env前提＋推奨shipグループ）で可。

  **§3 中/高/土台レーン（🟡中・🔴高・🧱土台 **または 🟢低でも要人力確認カテゴリ該当**）**
  - **c'. RLS/anon 機械監査（migration差分がある時だけ・軽め）**：差分に `supabase/migrations/` 追加/変更を含む場合のみ `node scripts/rls-audit.mjs --assert --json` をローカル(適用済)に対して実行。**🔴LEAK** があれば 📋に🔴で載せる（gateは止めない＝レビュー待ちには上げ人間判断に委ねる）。
  - **c''. 【Gemini二重レビュー】（🔴高・🧱土台のみ必須／🟡中は任意）**：`node scripts/independent-review.mjs "<diff範囲>" --json --ticket-id <pageId> --ticket-file <要件md>` を実行（Claude非依存の Gemini が `.kody/rules/` ＋ チケット要件で diff をレビュー＝**実装AIと別AIによる二重チェック**）。**findings／verdict／riskClass を 📋ダイジェストに追記**。**ブロッカー級（未accept critical / 🔴LEAK）は自分で修正を試行**し、緑化できたら再レビュー。**修正不能なら「⚠️要注意」を 📋に明示**（block はせず人間verdictに委ねる＝スクリプトが Notion callout＋通知）。`GEMINI_REVIEW_API_KEY` 未設定なら**スキップした旨を📋に明記**（gate扱いにしない）。
  - **d. 昇格**：`リスク`(🔴/🟡/🟢 Select)・`土台`(該当 Checkbox=true) を**プロパティに直接セット**し、本文に **📋レビューダイジェスト**（§ダイジェスト・c'/c''結果を含む）＋ **🧪人力チェック手順**（§人力チェック手順・**中/高/土台 または 要人力確認カテゴリ該当（🟢低含む）は必須**）を書いて、`AUTO_MERGE_TARGET`(=dev)へ**devマージ**（push しない）→ ステータス=**レビュー待ち**・🤖外す。**review→本番待ち 昇格はしない**（人間が行う）。※ここで書いた🧪手順は人間が本番待ちへ昇格した後もそのまま ship 前の人力確認に使う。
- **e. 本番に触れない**。dev 未push前提で積む。

### §緊急レーン（`優先順位=緊急` のみ・main派生で dev堆積から隔離）
通常レーン（dev基点→devマージ→レビュー待ち/本番待ち）と分け、**緊急だけは clean な main から分岐**して独立shipできる形にする。これにより「ドレイン徹底で dev のマージ待ちがどれだけ膨らんでも、緊急は巻き込まれず即shipできる」を担保する。

手順（毎ループ冒頭の緊急監視で検知したら、他の何より先に）:
1. **隔離ブランチ**：`git fetch origin {{PROD_BRANCH}}` → `git switch -c hotfix/<slug> origin/{{PROD_BRANCH}}`（**dev ではなく origin/{{PROD_BRANCH}} 基点**。作業中の dev 変更があれば commit/stash してから）。
2. **着手記録**：ステータス=`進行中`＋`🤖AI実行中`。
3. **実装**：通常と同じ品質。追加のみDDLは作成＋local適用（本番適用しない）。実装中に業務判断が要ると判明したら **要回答パーク**（hotfixブランチは破棄 or 残置メモ）し dev へ戻って次へ。
4. **ゲート**：§テストゲート（動的検出・全部緑）。migration差分あれば `rls-audit`。**🔴高/🧱土台は Gemini二重レビュー必須**（緊急でも品質は落とさない）。緑にできなければ blocked（保留）にして dev へ戻る。
5. **コミットのみ（★pushしない）**：hotfixブランチに commit。**push / PR / Merge はしない**（＝本番ゲートは人＝/ship）。
6. **着地**：📋レビューダイジェスト＋🧪人力チェック手順を本文に書き、**リスク不問で `本番待ち`**（レビュー待ちを経由しない＝速さ優先。🔴高の本番レビューは /ship 時に人が実施）。📋に「**緊急hotfix・ブランチ=`hotfix/<slug>`（main派生・未push）**」を明記（/ship が拾えるように）。
7. **🚨緊急通知**：`notify-humanball.mjs --kind 要対応`（または緊急用文面）で「緊急hotfix `<タスク>` ゲート緑・本番待ち。/ship で push→{{PROD_BRANCH}}宛PR→Merge を」。
8. **戻る**：`git switch dev`（or 元の作業ブランチ）して通常ループを継続（緊急は1件ずつ。複数あれば都度この手順）。

> /ship 側の対応：本番待ちのうち **ブランチが `hotfix/*`（main派生）** のものは、dev経由ではなく **その hotfix ブランチを push → {{PROD_BRANCH}}宛PR → Merge** で独立shipする（dev堆積と無関係に出せる）。通常の本番待ち（dev上）は従来どおり dev→PR→main。

### 5. （任意）preview/staging 再検証
preview/staging が構成済みならそこへデプロイし同じスイートを再実行（**無ければスキップ**）。

### 6. 停止条件
- **①枯渇（＝正常終了・唯一の通常停止）**：**未整理=0 かつ 要件定義済みに「実装可能な残り」が無い**（全件が レビュー待ち/本番待ち/要回答/blocked(保留)/完了 に収束）。**レビュー待ち・本番待ち・マージ待ち（dev堆積）は何件溜まっても停止理由にしない**（dev未push＝本番影響ゼロ。マージ待ち膨張のリスクは「緊急の main派生独立ship」で吸収済み）。**ディスパッチャの閾値（レビューバッチ等）は“通知トリガー”であって停止条件ではない**——閾値超えでも /run は止めず要件定義済みを掘り続ける。
- **②本番マージ要求**：人間の push/ship/昇格 を要する地点。CCは実行せず通知して待つ。
- **③異常（回路遮断）**：**連続前進ゼロ K=3**／**壁時計 MAX_WALL=180分**／**プラン枠で落ちた**。停止時は**枯渇か異常かを明記**。API課金へ勝手に流れない。
- 上記以外は**ノンストップ**。**「マージ待ち/レビュー待ちが多いから一旦区切る」は禁止**（①の枯渇まで掘り続ける）。
- **【厳守・止まる理由にしてはいけないもの】**：以下を理由に停止・中断・「続けますか？」と確認するのは**禁止**。①〜③の正規の停止条件**以外では必ず次の1件へ自動で進む**：
  - 「セッション/会話が長い」「出力(トークン)が多い」「ターンが長くなった」→ **止めない**（長さは停止条件ではない）。
  - 「次の案件が大きい/重い」→ 大きくても着手する（重い🔴高は§3の手厚い工数で。業務判断が要ると判明したら要回答パークして次へ）。
  - **【最重要】「リスクが高い(🔴高/🧱土台)」「重要度が高い」「要人力確認カテゴリ該当」「金額/認証/データ統合に触れる」→ これらを理由に停止・チェックポイント・確認するのは禁止**。**そのまま実装してレビュー待ち/本番待ちまで流す**。リスクの担保は「実装を止めること」ではなく **/review の徹底テスト（影響範囲マップの全参照をbreadthで確認・自動E2E追加・本番スモーク）** で行う（2026-07-02 明文化）。＝「🔴高だから慎重に一旦止める/セッション末尾だから後で」は禁止。§3の手厚い工数（Explore影響範囲マップ・edge E2E・Gemini二重レビュー）を**その場で実施して前進**し、人ゲート（レビュー承認/ship）まで自動で持っていく。唯一パークするのは §6②要回答（業務判断が要る intent-gap）だけで、**リスクの高さは要回答の理由にならない**。
  - 「レビュー待ち/本番待ち/マージ待ち（dev堆積）が溜まった/区切りが良い」→ §6①の枯渇まで止めない（堆積は正常・本番影響ゼロ。膨張リスクは緊急のmain派生独立shipで吸収済み）。
  - **人間に「続けていいか」を聞かない**（[[feedback_run_no_confirmation]]）。1ユニットごとの1行報告はしてよいが、承認は求めず即次へ。
  - 報告で「一旦区切る/ここまで」と書くのは、①枯渇 か ②本番ゲート か ③異常 の時だけ。それ以外で区切らない。

## リスク・エピック自動判定（§2 棚卸し・§3d 付与の判定源）
- **`リスク`(Select)**：**🔴高**=schema/migration・共有util・データモデル・認証/RLS・課金・外部送信／**🟡中**=複数画面・複数ファイルの機能追加／**🟢低**=自己完結UI・文言・単一トグル・設定値。
- **`土台`(Checkbox=true)**：後続の未着手/要件定義済みが**この成果物に依存**するなら true（後続件名/page_idを📋の「依存」に記す）。
- **`エピック`(Select)**：ship単位。近い画面・機能群・既存エピックに寄せる（※エピック例はプロジェクト固有＝**Notion 側の実エピック値**に合わせる。代表例は各リポ CLAUDE.md「Pipeline設定」参照）。一意に決まらなければ近接既存エピックへ。**どの既存エピックにも収まらない時だけ新規エピックを作成**してよい（粒度＝ship単位。単発タスクのための乱立は避け、まず既存への寄せを優先）。
- **`優先順位`**：本文の緊急度語・依存・ユーザー直指示から（緊急＝障害/業務停止級、高＝近接の本命機能、中＝通常追加、低＝軽微）。
- いずれも**Select/Checkbox プロパティ値に直接セット**（本文・独自タグで代替しない）。判断が割れる業務的なものだけ §2(a) 要回答へ。

## リスク別 工数（effort）スケーリング（§3 実装の深さをリスクで変える）
**狙い**：🟢低は軽くして初速UP・🔴高は手厚く堅く出す。リスク階層に応じて CC の作業の深さを下表で変える（過剰実装も検証不足も避ける）。

| 観点 | 🟢低 | 🟡中 | 🔴高/🧱土台 |
|---|---|---|---|
| **着手前の調査** | なし（直接実装） | 該当ファイル確認 | **Exploreで全消費箇所をマップ**（直し漏れ/footgun回避） |
| **自動テスト** | build/typecheck のみ | build＋関連E2E＋**保存→再読込の永続/回帰1本** | build＋**境界値・二重計上等のedge E2E**＋全消費箇所green |
| **Gemini二重レビュー** | なし | 任意(runs=1) | **必須(runs=2〜3 union)** |
| **RLS/anon監査** | なし | migration差分時 | **必ず実行** |
| **CCの思考の深さ** | 速く決める | 標準 | **熟考・検算・複数視点で確認** |
| 人力レビュー(§/review) | ノールック（本番待ち直行） | 要点ステップ | **詳細ステップ＋デモデータ用意＋⚠承知確認** |
| ship(§/ship) | ノールック可 | 本番軽スモーク | **本番スモーク必須** |

**最低ライン（リスク不問）**：機能追加は最低でも「**保存→再読込で永続するか**」を1本は確認する（UI追加だけでDB制約/保存漏れを見逃さないため。例: enum/区分追加は CHECK制約・保存経路まで）。

**データ統合・マージ・付け替え・一括更新は「統合後の結果」まで検証する（2026-07-02追加）**：複数レコードを1つにまとめる/移す変更は、**マージ元だけでなくマージ先の集計・関連・参照が正しく統合されたか**を自動テスト＋人力テストで必ず確認する（例：現場マージ後に**現場別集計が統合先へ合算されているか**・孤児レコード/二重計上が無いか）。「マージ操作が成功した」で終わらせない＝統合の下流（集計・レポート・関連テーブル）が壊れていないかまで見る。この種の変更は要人力確認カテゴリに該当＝§3レビュー待ちへ。

### 隠れリスク発見＝工数昇格ルール（重要）
分類は着手前の見立てに過ぎない。**実装中に下記を発見したら、その場でリスクを上げて工数も上げる**（or §2(a)要回答パーク）。事故った"軽い実装"を防ぐ最後の砦：
※**昇格は「実装に着手して実コードで判明した時だけ」**。着手前の見立てで🟡中を🔴高扱いして**スキップするのは禁止**（盤面の分類を尊重し、低リスクから順に捌く＝§3-4 厳守）。判明したら下記で昇格する：
- **金額/原価/集計に触れる**（二重計上・按分・retroactive変更）→ 🔴高扱い＝edge E2E＋Gemini＋（業務判断なら）要回答。
- **データモデル/スキーマ前提が想定と違う**（リレーション無い・CHECK制約・enum固定 等）→ migration要否を再判定し🟡→🔴へ。
- **共有util/複数消費箇所**（金額/集計/外部送信を複数箇所が消費する横断utility ※実体はプロジェクト固有の共有モジュール＝CLAUDE.md「Pipeline設定」APP_LAYOUT 参照）→ チェックリスト全消費箇所を直す（直し漏れ防止）。
- **anon露出/認証/RLS/外部送信** に触れると判明 → 🔴高＋RLS監査＋accepted.yml追跡。
昇格したら Notion の `リスク` プロパティも上げ、📋ダイジェストに「着手時に判明し昇格」と1行残す。

## 自動テストゲートの動的検出（ハードコードしない＝有るものを全部）
**特定テストを名指ししない。実在ゲートを毎回スキャンして全実行**。命名規約に沿えば将来分も自動で取り込む。
> **★カバレッジのギャップは埋める（既存テスト実行だけで満足しない・2026-07-02）**：要人力確認カテゴリ（特にデータ統合/マージ/集計/データモデル）は、**影響範囲マップの主要参照を検証する自動テストが無ければ新規に1本足してから**ゲートを通す。「既存テストが全緑」＝「今回の変更の影響先が検証された」ではない（今回の現場マージは既存テスト全緑でも集計統合を誰も検証していなかった）。追加した/できなかったテストは📋に明記。
1. **typecheck / build**: root と各 `apps/*/package.json` の scripts から `typecheck` / `build` を検出して全実行（{{APP_LAYOUT}}・具体コマンドは **CLAUDE.md「Pipeline設定」の TYPECHECK / BUILD** に従う。pnpm monorepo なら `pnpm -r typecheck` / `pnpm -r build` 等）。
2. **テストscript**: root と `apps/*` の scripts のうち **`/^test/` に一致する全部**（{{TEST}}・`test` / `test:e2e` / `test:unit` / 将来の `test:visual` 等。**TEST が none＝テストscriptが無ければこの段はスキップ**）。
3. **Playwright**: `playwright.config.*` があれば定義された **projects を全部**（{{PLAYWRIGHT_PROJECTS}}・未設定/playwright.config 無しなら全spec or **スキップ**）。関係 spec は緑を明示。
4. DB/E2E を要するものは §0 プリフライトで **local を保証**してから実行。
- いずれか緑でない→ そのアイテムは **blocked**（落ちた script 名と要約を記録）。「3回直しても緑にできない」も blocked。
- **既知の事前失敗（本変更と無関係）は回帰判定から除外**してよい（除外したものは📋に明記）。

## Gemini二重レビュー（別AIによる独立二重チェック）の呼び出し設計
- 実体は `scripts/independent-review.mjs`。**実装はClaude／レビューはGemini**の二重構成。**`GEMINI_REVIEW_API_KEY` を HTTPヘッダ `x-goog-api-key` でクライアントへ明示的に渡す**（`GEMINI_REVIEW_MODEL` も .env 由来。**SDK自動検出に依存しない**）。
- 入力＝`.kody/rules/*.md`（ルール）＋ `--ticket-file`（要件md：意図照合）＋ `git diff <範囲>`。出力＝JSON（`findings[]` / `accessRuleEnforced` / `verdict` / `riskClass` / `accepted`）。`.kody/accepted.yml` で既知許容を block 除外。
- 呼び出し（🔴高・🧱土台で必須）:
  ```bash
  node scripts/independent-review.mjs "<diff範囲>" --json \
    --ticket-id <NotionページID> --ticket-file <要件mdパス>   # /run が --dry-run の時は末尾に --dry-run
  ```
  - `<diff範囲>` は当該レーン作業の差分（例 `dev...HEAD` ＝コミット済み feature 差分。未コミットは先にコミット）。
  - **頑健化が要る重要差分は `--runs 2`**（2回実行して findings を union＝非決定性の保険＝文字通りの“二重”）。
- 結果の扱い：**未accept critical / 🔴LEAK＝ブロッカー級は自分で修正→再レビュー**。緑化できたら昇格。**修正不能なら 📋に「⚠️要注意（要人間verdict）」を明示**してレビュー待ちへ（スクリプトが Notion callout＋通知を自動で立てる）。`verdict` は📋に必ず転記。

## ディスパッチャ（決断の瞬間だけ通知・`scripts/dispatcher.mjs`）
- /run 末尾から呼ぶ＋単体実行可（`node scripts/dispatcher.mjs [--dry-run]`）。判定材料は既存プロパティのみ（ステータス/リスク/土台/優先度/案件/エピック）。配信は既存 LINE human-in-the-loop（`notify-humanball.mjs`）。
- **dedup 必須**：`scripts/.dispatcher-state.json`（git管理外・キー=`page_id:status`）。毎実行プルーン（status変化/消滅キー除去）→差し戻し再入で再通知。**per-item スパム禁止＝1ダイジェスト集約**（🧱土台の即時のみ個別）。**初回統合**：プルーン後 state が空なら 🧱土台＋レビュー待ち＋要回答 を1通に統合して1回だけ送信。
- **緩急**: 🧱土台/🔴=即時、🟡/🟢=バッチ。

トリガー（**いずれも 要約＋本文リンクを載せ・🧪人力チェック手順の全文は本文**）:
1. **🧱土台がレビュー待ち入り** →【即】「🧱土台『X』レビュー待ち・後続N件依存・今レビュー推奨。🧪人力チェック手順は本文: <page url>」（個別なので本文リンクを直付け）。
2. **レビューバッチ準備**（土台以外のレビュー待ちが閾値超 or 🔴≥3 or max-age flush）→「レビュー可: 🔴a/🟡b/🟢c、要対応ビューへ（各本文に🧪人力チェック手順）」。
3. **エピックship可**（あるエピックの actionable が**全て本番待ち**）→【まとめ】「📦エピック『○○』ship可: N件(🔴a/🧱b/🟢c)。**🔴高・🧱土台は本番レビューしてからship／🟢はノールック可**。各本文の🧪手順でスモーク・env前提を本番確認」。
4. **要回答（業務判断）** → まとめて「判断待ちN件: ①…②…（A/B/C＋推奨）」。
5. **緊急検知** → 即時。 **異常/停止** →「drain停止: 理由・残件」。

- **レビュー待ち→本番待ち 昇格（=レビュー承認）も人間**（CCは中高土台を昇格させない）。

- `ship.md` は従来どおり **本番待ち のみ**拾う（未レビューの中高は構造上ここに来ない）。エピックship時、🟢はノールック・🔴/🧱は本番レビューしてから。

## DRY_RUN 動作確認手順（副作用ゼロ）
1. 未整理を数件置く（材料十分な1件・本文空/曖昧な1件）。要件定義済みも数件（**🟢低を1件・🔴高を1件・🧱土台を1件・緊急を1件**）。
2. `/run --dry-run` → 提示のみ：盤面／実装予定の**優先順位順（緊急>高>中>低・同順位は作成古い順）**／「(b)→要件化＋4プロパティ付与・(a)→要回答」／**緊急プリエンプト予定**／要件定義済み各件の**レーン振り分け**（🟢低かつ要人力確認非該当→自動マージ→本番待ち予定／🟢低でも要人力確認該当・🟡🔴🧱→レビュー待ち予定・🔴高🧱土台は**Gemini二重レビュー予定**）／各件の**要人力確認カテゴリ該当有無と理由**／実行予定の**自動ゲート一覧**／付与予定 リスク/土台／**ディスパッチャ通知予定**。Notion/コード/通知/テスト/マージは一切しない。
3. `node scripts/dispatcher.mjs --dry-run` → 現盤面から各トリガーの発火予定だけ列挙（送信せず・state更新なし）。エピックが全て本番待ちなら「📦エピックship可 N件(🔴/🧱/🟢)」の発火予定が出る。
4. 実行（flagなし）で：🟢低が**本番待ち**へ・🔴高/🧱土台が**レビュー待ち**へ積まれ、🧱土台に**即時通知**・🟢leaf は**バッチ**に溜まること、エピック全完了で**📦エピックship通知**が出ること、再 `/run` で**再通知されない**（dedup）ことを確認。

## 厳守
- 本番反映（push / deploy / db push / ship）と **レビュー待ち→本番待ち昇格** は人間の明示OKまで実行しない。それ以外は自走。
- **停止は §6 の3つ（要件定義済み枯渇／本番ゲート／異常）だけ**。**レビュー待ち・本番待ち・マージ待ち（dev堆積）が何件溜まっても止めない**（ディスパッチャ閾値は通知であって停止条件ではない）。「堆積したから区切る」は禁止。
- **リスク/重要度の高さで止めない（最重要）**：🔴高・🧱土台・要人力確認カテゴリ・金額/認証/データ統合 でも、停止・チェックポイント・確認せず**その場で§3の手厚い工数（影響範囲マップ・edge E2E・Gemini）を実施して実装し、レビュー待ち/本番待ちまで流す**。リスク担保は「止めること」でなく **/review の徹底テスト**で行う。パークは §6②要回答（業務判断のintent-gap）のみ＝リスクの高さは要回答理由にならない。
- **実装順は優先順位順（緊急>高>中>低・同順位は作成日古い順）**。リスクは順序に関与させない（レーン分岐と工数スケーリング用）。上位優先度を残して下位に先行しない。
- **緊急（`優先順位=緊急`）は main派生（origin/{{PROD_BRANCH}}基点）hotfix ブランチで実装**し、ゲート緑→**本番待ち（レビュー待ちを経由しない）＋🚨緊急通知**。**push/PR/Merge はしない**（人＝/ship）。通常チケットは従来どおり dev基点。
- **低リスクは dev自動マージ→本番待ち（レビュー不要）／中高土台は レビュー待ち止まり**。**ただし🟢低でも要人力確認カテゴリ（金額/集計/データ統合・マージ/migration/認証・RLS/外部送信）に触れる差分は §3レビュー待ちへ回し人力テスト必須**（自動判定・実差分で）。リスク＋要人力確認のレーン分岐を必ず守る。
- **中/高/土台 または 🟢低×要人力確認カテゴリ該当 をレビュー待ち/本番待ちに置くときは 🧪人力チェック手順 を本文に必ず生成**（§人力チェック手順・🔴🧱=詳細/🟡=要点・要人力確認該当は「統合/集計/金額の目視」を必ず1ステップ）。通知は要約＋本文リンク・全文は本文。ミスが波及しない純🟢低のみ🧪不要。
- 人ボール化（要回答）は人の回答まで再着手しない（同じ問いでループしない）。
- リスク/土台/エピック/優先度は必ず **Select/Checkbox プロパティ値**に入れる（本文・独自タグで代替しない）。
- 「🤖AI実行中」タグは着手で付け、本番待ち/レビュー待ち/要回答/blocked/中断で必ず外す。原則同時1件。
- `--dry-run` は副作用ゼロ。いつでも中断可能。
