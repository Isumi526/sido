# /run — ステータス駆動の自走ステートマシン

CLAUDE.md「自走ポリシー」に従い、**バックログのステータスを唯一の真実**として、自案件（`.env` の `BACKLOG_PROJECT_ID`）の各タスクを**ステータスに応じて1フェーズずつ前進**させる単一ループ。人間ゲートに当たったタスクは止め、**ループは次のタスクへ継続**（全体は止めない）。終了条件（未整理=0／上限到達）まで回る。

引数: `$ARGUMENTS`
- `--dry-run` … **何件をどのフェーズに動かすか提示のみ**（Notion更新・コード変更・通知・ブランチ作成を一切しない）。**初回・仕様変更後は必ずこれで確認**。
- `--advance` … 着手ゲートを開ける（`AUTO_ADVANCE=true`）。要件定義済み→実装着手まで自動で進む。**未指定なら既定 false**（要件定義済みで止めて次へ）。
- `--max-items=N` … 1回の/runで前進させるタスク数の上限（既定 20）。
- `--max-minutes=M` … 経過時間の上限（既定 30）。
到達したら打ち切って続きは再/run。

**人ボールを出す時は CLAUDE.md「人ボール報告の必須フォーマット（結論ファースト）」に従う**：画面は「🙋 [種別] タスク名／【決めてほしいこと】／【選択肢】A/B/C／【推奨】X」。`--detail` は1〜2行に圧縮。

---

## 0. 開始時の自己修復（ループ前に1回）
自案件のタスクで **「🤖AI実行中」タグが付いたまま残っているもの**があれば**全部外す**（前回クラッシュの取り残しクリア）。Notion API で「タグ」（複数選択）を読み、`🤖AI実行中`（絵文字込み・完全一致）を**既存配列から除いて**書き戻す。`--dry-run` 時は「外す対象」を列挙のみ（書き換えない）。

## 1. 盤面取得（全ステータス）
`node scripts/next-target.mjs --board` で**自案件の全ステータスのスナップショット**を取得（案件で絞り済み・notion-searchは使わない）。各タスクの ステータス / 優先順位 / タグ / URL が得られる。

---

## ステータス → フェーズ 対応表（ステートマシンの核）

| ステータス | 区分 | /run が行う1フェーズ前進 | 遷移先 | ゲート |
|---|---|---|---|---|
| **未整理** Inbox | 仕分け | 本文を判定。(a) 意図/業務判断が必要 or 本文が空/不足 → **要回答**化（質問＋案）。(b) 情報十分 → 薄いストーリー+AC+scope に**要件化** | (a) 要回答 / (b) 要件定義済み | (a) で停止 |
| **要回答** | 🛑GATE | 何もしない（人の回答待ち・同じ問いで再質問しない） | 要回答（不変） | 停止済み |
| **要件定義済み** Ready | 着手判断 | `AUTO_ADVANCE=true` → 着手（進行中へ）。`false`（既定）→ **動かさず**「着手可能」として記録し停止 | 進行中 / 要件定義済み | `!AUTO_ADVANCE` で停止 |
| **進行中** | 実装 | **CCの作業跡（featureブランチ/設計書/🤖タグ）がある進行中のみ**再開し、設計→実装→E2E緑→devマージまで完了させ**レビュー待ち**へ。跡が無い＝人手の進行中は**触らない** | レビュー待ち | →ゲートで停止 |
| **レビュー待ち** | 🛑GATE | 何もしない（人のレビュー待ち。レビュー通過＝人が本番待ちへ移す） | レビュー待ち（不変） | 停止済み |
| **本番待ち** | 🛑GATE | 何もしない（`/ship` での本番反映承認待ち） | 本番待ち（不変） | 停止済み |
| **保留 / 完了** | 終端 | 対象外（触らない） | 不変 | — |

**着手＝Notionを「進行中」に更新＋「🤖AI実行中」タグ付与（既存タグ保持して追加）**。**レビュー待ちへ上げる時にタグを外す**。中断・人ボール化の時も必ず外す（付けっぱなし厳禁・原則同時1件）。

## 2. メインループ（擬似コード）

```
board = next-target.mjs --board
processed = ∅                      # 1ループ内で同一 page_id を二度処理しない
movedToGate = []                   # この実行でゲートに入れた件（通知対象）
t0 = now

for item in priorityOrder(board):  # 中断中(進行中/レビュー待ち) → 要件定義済み → 未整理 の順、各群内は 緊急>高>中>低, 同率は作成日古い順
    if item.id in processed: continue
    if reached(MAX_ITEMS) or elapsed() >= MAX_MINUTES: break
    if item.status in {要回答, レビュー待ち, 本番待ち, 保留, 完了}:
        continue                   # 既にゲート/終端で停止済み。再通知しない（多重発火させない）
    processed.add(item.id)

    switch item.status:
      case 進行中:                 # 実装途中の再開。前進のみ（多重発火させない）
        if 作業跡なし(featureブランチ/設計書/🤖タグ いずれも無い): continue   # 人手の進行中は触らない
        if DRY_RUN: plan(item, "進行中 → レビュー待ち（実装を完了させる）"); continue
        実装を完了 → E2E緑 → devマージ
        if E2E が3回直しても緑にならない:
            → 要対応化（記録）＋タグ外す → notify(要対応) → continue
        set 本番待ちではなく → set レビュー待ち；タグ外す；movedToGate += (item, レビュー待ち)

      case 要件定義済み:           # 着手ゲート
        if not AUTO_ADVANCE:
            if DRY_RUN: plan(item, "要件定義済み（着手可能・AUTO_ADVANCEで実装）"); continue
            記録「着手可能」；continue            # 動かさない＝停止（次へ）
        # AUTO_ADVANCE=true
        if DRY_RUN: plan(item, "要件定義済み → 進行中 → …(実装) → レビュー待ち"); continue
        set 進行中＋🤖タグ → 設計→実装→E2E緑→devマージ
        if E2E 3回で緑にならない: → 要対応化＋タグ外す → notify(要対応) → continue
        if 途中で(A)意図必須が判明: → 要回答化＋タグ外す → notify(要回答) → continue
        set レビュー待ち；タグ外す；movedToGate += (item, レビュー待ち)

      case 未整理:                 # 仕分け
        判定 = 本文に要件化の材料があるか
        if 材料なし / 意図・業務判断が必要:           # (a)
            if DRY_RUN: plan(item, "未整理 → 要回答（質問＋案）"); continue
            set 要回答；movedToGate += (item, 要回答)
        else:                                          # (b)
            if DRY_RUN: plan(item, "未整理 → 要件定義済み（要件化）"); continue
            薄いストーリー+AC+scope を生成 → set 要件定義済み
            # 要件化中に(A)意図必須と判明したら (a) に格下げ → 要回答化 → notify(要回答)
        # 未整理を1件処理したら盤面が変わるので必ずループ先頭へ戻る（再取得 or in-memory更新）

    1行サマリ（非ブロック）を出す

# ループ後：この実行でゲートに入れた件を通知（到達時1通・既存パーク分は通知しない）
for (item, gate) in movedToGate: notifyGate(gate, item)
report(盤面サマリ)
```

`priorityOrder` の群順は CLAUDE.md「/run の優先順位決定」と一致（中断再開を最優先 → Ready → Inbox）。

## 3. ゲート通知（要回答 / レビュー待ち / 本番待ち）
**この実行で /run がそのゲートに“入れた”時だけ 1通**（既にそのステータスで止まっていた既存分は再通知しない＝多重発火防止）。**PRマージ承認（/ship の `ship承認`）とは別 kind・別文面で混線させない**。

```bash
# 要回答（未整理→要回答 / 実装中に意図必須が判明）
node scripts/notify-humanball.mjs --kind 要回答     --task "<タスク名>" --detail "<決めてほしいこと＋A/B/C＋推奨を1〜2行>" [--url "<セッションurl>"]
# レビュー待ち（実装完了・人レビュー待ち）
node scripts/notify-humanball.mjs --kind レビュー待ち --task "<タスク名>" --detail "実装完了・devマージ済み。レビューOKなら本番待ちへ。<PR/ブランチ>" [--url "<セッションurl>"]
# 本番待ち（/run が本番待ちに上げた場合のみ。通常はレビュー通過＝人が移すのでまれ）
node scripts/notify-humanball.mjs --kind 本番待ち   --task "<タスク名>" --detail "本番待ち。/ship で反映を" [--url "<セッションurl>"]
# E2Eが直らない等
node scripts/notify-humanball.mjs --kind 要対応     --task "<タスク名>" --detail "<失敗理由/エラー要約>" [--url "<セッションurl>"]
```
- `ship承認`（PRごとの「今Mergeして」）は **/ship 専用**。/run は使わない。
- 1停止で複数承認をまとめる場合のみ通知1回（連投しない）。失敗しても best-effort で続行。

## 4. 冪等・安全
- 各フェーズは**前段が完了したタスクだけ**前進（ステータスがその一歩手前の時のみ動かす＝多重発火しない）。
- **1ループ内で同一 page_id を二度処理しない**（`processed`）。
- ゲート/終端（要回答・レビュー待ち・本番待ち・保留・完了）のタスクは**触らない・再通知しない**。
- 自プロジェクト（`BACKLOG_PROJECT_ID`）の page_id 以外は対象外（`--board` が案件で絞済み）。
- **migration 不要**（ステータス選択肢は追加済み）。
- 本番デプロイ / main マージ / db push は絶対しない（/ship 専用）。migration は作成のみ。

## 5. DRY_RUN（初回・仕様変更後は必須）
`/run --dry-run`：盤面を取得し、各タスクが**どのフェーズへ動く“はず”か**だけを表で提示。**Notion更新・コード変更・ブランチ・通知を一切しない**。
出力例：
```
[DRY] 盤面: 未整理30 / 要回答0 / 要件定義済み3 / 進行中2 / レビュー待ち0 / 本番待ち1 / 保留3 / 完了45
[DRY] 自己修復: 🤖AI実行中タグ残り = (なし)
[DRY] 進行中(2)  → レビュー待ち（実装完了させる）: «タスクA», «タスクB»
[DRY] 要件定義済み(3) → AUTO_ADVANCE=false のため停止（着手可能）: «…»×3   # --advance時は「→レビュー待ちまで」
[DRY] 未整理(30) → 先頭から判定: (b)要件化 «…» / (a)要回答 «…»  ※実際は1件ずつ盤面再取得
[DRY] ゲート到達で送る通知: レビュー待ち×2（--advance時）, 要回答×N
[DRY] 上限: MAX_ITEMS=20 / MAX_MINUTES=30
```

### DRY_RUN 検証手順（ゲートで正しく止まるか）
1. **未整理を数件用意**：材料十分な1件・本文が空/曖昧な1件をバックログに置く。
2. `/run --dry-run` → 「(b)→要件定義済み」「(a)→要回答」に振り分けられ、**実装には進まない**ことを確認。
3. `/run`（flagなし）→ (b)は要件定義済みへ・(a)は要回答化＋通知、**要件定義済みは着手せず停止**を確認。Notionと通知1通ずつを目視。
4. `/run --advance` → 要件定義済み/進行中が実装され**レビュー待ちで停止**・通知1通を確認。レビュー待ち/本番待ちは**触られない**ことを確認。
5. 各ゲート（要回答/レビュー待ち/本番待ち）のタスクが**再/runで再通知されない**ことを確認（冪等）。

## 6. 停止条件（ここで初めて止まりサマリ報告）
- **未整理=0 かつ 着手対象なし**（Ready無し or AUTO_ADVANCE=false で全Ready停止、進行中も無し）→ 停止して盤面サマリ報告。
- `MAX_ITEMS` / `MAX_MINUTES` 到達 → 打ち切り、続きは再/run。
- 未整理が **(a)意図必須ばかり** → 各々を要回答化＋通知し、全処理後に停止（人の確定待ち）。「未着手だから検知できない」状態は作らない。
- 本番待ちがあり他に actionable 無し → 「本番待ちN件。/ship で承認を」。

## 厳守
- 本番デプロイ / main マージ / db push は絶対しない（/ship 専用）。migration は作成のみ。
- 人ボール化（要回答/要対応）したタスクは人の回答が入るまで再着手しない（同じ問いでループしない）。
- 「🤖AI実行中」タグは“CCが実装中”の印。着手で付け、レビュー待ち/人ボール/中断で必ず外す。原則同時1件。
- ゲート通知は `ship承認` を使わない（/ship と混線させない）。`--dry-run` は副作用ゼロ。
- いつでも中断可能。
